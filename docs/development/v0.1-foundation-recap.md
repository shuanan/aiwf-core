# AIWF Core v0.1 Foundation Recap

Status: historical snapshot of the v0.1 foundation milestone. Not a live status report.

This document summarizes the `aiwf-core` foundation as of the v0.1 foundation milestone (after the initial standalone extraction and validation hardening). For current state, read the repo directly (see `docs/development/sync-anchor.md`).

## Foundation milestone state

As of the v0.1 foundation milestone, AIWF Core was a standalone portable governance library.

It had:

- Standalone public repository.
- Architecture / kernel / registry / adapter / worker / control-plane skeleton.
- Local validation script.
- GitHub Actions validation.
- YAML parse validation.
- Internal registry / release reference validation.
- Minimal downstream adapter example.
- Example adapter structure validation.
- Declaration Integrity Gate.
- Quickstart documentation.
- Foundation checklist.

## Validation floor at the foundation milestone

At the v0.1 foundation milestone, the validation floor included checks that:

- Required files exist.
- Forbidden default runtime surfaces are absent.
- `.claude/settings.json` is absent.
- Runtime hooks are not installed by default.
- YAML parses with PyYAML in CI.
- Registry capability `location` paths exist.
- Approved capabilities include `authority_boundary.can` and `authority_boundary.cannot`.
- Release manifest capability IDs exist in the registry.
- Minimal example adapter remains a safe draft.
- Declaration mismatches include remediation guidance.

This list reflects the foundation milestone and is not kept current. For the
current validation result, run:

```bash
bash scripts/validate-aiwf-core.sh
```

## Important design correction

The `slack-notify` mismatch produced the Declaration Integrity Gate.

Rule:

> Validators detect mismatch. Humans decide intent. AI may draft a patch only after the decision is clear.

This prevents validators or AI agents from silently choosing whether to remove, register, promote, or adopt a capability.

## What is intentionally not included

The foundation does not include:

- Slack automation.
- Linear automation.
- Notion automation.
- Langfuse integration.
- Codex execution.
- Claude execution.
- Claude `-p`.
- Claude Code hooks.
- `.claude/settings.json`.
- Worker auto-run.
- Paid worker execution.
- Auto-fix.
- Auto-merge.
- Production deployment.
- Real downstream adoption.

## Current safe boundary

AIWF Core v0.1 foundation is documentation, schemas, examples, and validation only.

It is safe to use as a reference library and adapter template source.

It is not yet a workflow runner, automation framework, hook pack, or downstream adoption package.

## Next safe increments

Recommended next increments, in order:

1. Add a machine-readable release readiness checklist.
2. Add registry hygiene validation for unregistered artifacts.
3. Add schema examples for adapter / worker profiles / release manifest.
4. Prepare one notify/manual downstream pilot.
5. Only after pilot evidence, consider optional control-plane integration.

Do not start with Slack / Linear / Codex / Claude automation.
